# 工具使用策略

本文档不重复工具 schema，只记录 ZBot 在本环境下的工具选择策略和本地经验。

## 通用原则

- 能直接从文件得到答案时，优先读文件，不先联网。
- 能用更小、更安全的工具完成时，不优先使用更重的工具。
- 多步任务优先先观察、再定位、再修改、最后验证。

## 文件工具

### read_file

- 用于读取源码、配置、文档和大文件片段。
- 大文件优先分页读取，不一次读全。

### edit_file

- 优先用于局部修改。
- 当只改一个片段时，不要整文件重写。

### write_file

- 适用于新建文件或重写整个小文件。
- 覆盖写入前先确认这确实是预期行为。

## exec

- 优先用于观察性命令、搜索、测试、构建、状态检查。
- 对删除、覆盖、批量移动等高风险命令保持谨慎。
- 输出可能被截断；当结果不完整时，应进一步缩小范围重新查询。

## web_search 与 web_fetch

- 需要最新外部信息时，先 `web_search` 找候选来源，再 `web_fetch` 读取一手页面。
- 不要用网页搜索代替本地代码阅读。
- 外部资料只用于补足本地没有的事实。

## cron

- 用于定时提醒、周期检查、一次性定时执行。
- 不要把普通待办、会话内临时计划误建成 cron。
- 对用户说“我会记住”的事项，如果需要准时触发，应使用 cron 而不是只写记忆。

## create_sub_agent

- 只在复杂任务确实具备并行拆解价值时使用。
- 不要因为任务“看起来大”就创建子 Agent；只有多个子任务互不阻塞、可以并行、最终只需汇总时才使用。
- 每个子任务必须有明确输入范围、执行目标和预期输出。
- 工具输入使用 `subtasks` 对象列表；每个对象必须包含 `subtask_id`、`task`、`expected_result`。
- `create_sub_agent` 必须单独一轮调用，不要和 read_file、exec、web_search 等普通工具放在同一轮。
- 不适合顺序推理、连续调试、统一写作风格控制、需要子任务频繁协商的任务。
- 子 Agent 是一次性执行单元，不负责用户对话、长期记忆、会话归档或最终决策。
- 主 Agent 必须验证、整合子 Agent 返回的结果，并对最终回答负责。

## MCP 与技能

- 内置工具不足以完成任务时，再考虑 MCP。
- 某项能力已有技能说明时，先遵循技能文档，再补充本地特例。
